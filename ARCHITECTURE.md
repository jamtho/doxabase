# DoxaBase Architecture

This document orients agents and humans working in the repository: what
exists now, where the boundaries are, and which properties are deliberate.

## Purpose

DoxaBase is a local RDF memory capsule for data projects. The library
contains no LLM and does not decide semantics: agents supply semantic
judgement; DoxaBase supplies storage, parsing, named-graph mechanics,
bounded retrieval, staged change, privacy scanning, and explicit validation.

## Repository Map

- `doxabase/core/` — the capsule engine, one module per subsystem
  (storage, search, entities, datasets, slices, profiles, query planning,
  observations, map authoring, staging, restage, recovery, exports/privacy,
  brief, validation, ...), composed into the public `DoxaBase` facade via
  mixins. No module should exceed the scoreboard's line ceiling.
- `doxabase/mcp_tools.py` — testable MCP-callable functions plus the
  kind/aspect dispatch tables; `doxabase/mcp_server.py` — thin FastMCP
  registrations only. Keep business logic out of the decorators.
- `doxabase/agent_docs.py` — registry for the docs under `docs/agent/`
  served through `doxabase.get_doc`.
- `docs/agent/` — agent reference docs (budgeted; `docs/agent/mcp-tools.md`
  is GENERATED by `tools/gen_docs.py`). `docs/journal/` — dated history,
  wave log, trial reports; never MCP-registered.
- `ontology/rc_core.ttl`, `ontology/rc_shapes.ttl` — immutable seed graphs.
- `examples/manifest-prototype-rc/` — bundled AIS/Polymarket fixtures;
  `examples/*.py` — maintained executable walkthroughs.
- `tools/` — `scoreboard.py` (budget ceilings; the only place ceilings are
  defined), `gate.sh` (the full pre-commit gate), `gen_docs.py`,
  `validate_rdf.py`, `install_smoke.py`.
- `doxabase_design_docs/` — design history; doc 07 is the active
  Distillation Program, doc 08 the authoritative MCP surface mapping.

## Runtime Model

`DoxaBase` owns one SQLite file. On creation it builds minimal tables,
registers the default graph roles, and seeds `base_ontology`/`base_shapes`
from packaged Turtle. Storage is intentionally direct: RDF terms as strings
in a `quads` table with kind markers (`uri`/`bnode`/`literal`), literal
datatype/language stored separately, RDFLib used for parsing and for
reconstructing graphs at validation time. Staged revisions store row-level
snapshots of their target graphs so apply checks can report exact triples
added/removed since staging. An FTS5 table (`literal_search`) is a
derivative index over literal objects, rebuilt after writes; the quads stay
the source of truth.

Graph roles: immutable `base_ontology`/`base_shapes`; mutable `ontology`,
`map`, `observations`, `patterns`, `evidence`, `shapes`, `history`. Logical
includes: `ontology` = `base_ontology + ontology`, `shapes` =
`base_shapes + shapes`. Ordinary imports reject writes to immutable seeds.

### Concurrency stance

A capsule is a **single-writer** store, by decision (recorded 2026-07-04,
program Phase 1.6):

- One process holds a writable `DoxaBase` per capsule file; the intended
  deployment is one MCP server (or one script) per capsule.
- Concurrent readers use `DoxaBase.open_readonly(path)` (SQLite `mode=ro`).
- A second writer is not corrupted-by-design (SQLite serializes at the file
  level) but can hit `database is locked`; DoxaBase does not retry, queue,
  or merge. Staged revisions plus export/import handoff bundles are the
  supported reconciliation mechanism.
- WAL mode is deliberately **off**: sidecar `-wal`/`-shm` files would break
  the "a capsule is one file you can copy or hand off" property. Revisit
  only if a deployment needs higher read concurrency under an active
  writer.

## MCP Surface

Exactly **25 tools** (enforced by the scoreboard), mapped in
`doxabase_design_docs/08-mcp-surface-v2.md`: orientation reads
(`get_doc`, `project_brief`, `graph_overview`, `search`, `list_entities`),
describers (`describe_resource` with aspects, `describe_dataset`,
`describe_query_context` with `plan_candidate=` drafting,
`describe_revision` with aspects, `list_revisions` with kinds),
`get_context_graph` (TriG-first slices), kind-dispatched recorders
(`record_observation`, `record_profile`, `record_map_fact`,
`record_pattern`, `record_claim_reconsideration`, `record_graph_revision`),
the staged family (`stage_revision` with kinds + `dry_run` planners,
`apply_staged_revision` with `dry_run` as the check,
`restage_staged_revision` str|list, `plan_staged_revision_recovery` with
sessions), export/import (`export_preflight`, `export_bundle`,
`import_bundle` — kind-dispatched), and `validate_graph`.

Conventions: kind/aspect-dispatched tools validate `spec` per kind and fail
with targeted errors naming the valid/missing fields; responses omit
null/empty (absent == null == empty); graph content travels as TriG, JSON
is the envelope; suggestions are ≤5 `{tool, args, reason}` hints, not
instructions. `docs/agent/mcp-tools.md` is generated from the registry and
dispatch tables — regenerate with `uv run python tools/gen_docs.py`;
`tests/test_gen_docs.py` fails the gate on drift.

The Python facade (`from doxabase import DoxaBase`) keeps finer-grained
methods than MCP where that helps library users (e.g.
`db.check_staged_revision_apply`, `db.draft_query_plan`,
`db.export_preflight(export_kind=...)`); the MCP doors fold these into
`dry_run`/`plan_candidate`/`kind` parameters. Run the server with
`doxabase-mcp --capsule path.sqlite` (stdio default) or
`uv run python -m doxabase.mcp_server`.

## Validation And Privacy

Parse validation always; SHACL is an explicit diagnostic
(`validate_graph`, scopes `map`/`ontology`/`patterns`/`shapes`/`all`),
never a mutation commit gate — though importable graph exports gate on
invalid live graphs by default. Shapes are open: they catch common `rc:`
authoring mistakes without rejecting client RDF. Privacy posture:
`export_preflight` before shareable artifacts; scanner-clean is a review
prompt, not export approval; exports are faithful, never auto-redacted;
secrets never belong in the graph (credential references and endpoint
profiles name runtime configuration instead).

## Budgets And The Gate

The Distillation Program (doc 07) made cost a tested property.
`tools/scoreboard.py` holds the only `BUDGETS` dict — tool count, schema
chars, module/test line ceilings, response sizes on the standard fixture
capsule, docs byte totals — and `tests/test_budgets.py` enforces the same
ceilings in every pytest run. Ceilings only ratchet toward the end-state
targets; loosening requires James. The full pre-commit gate is
`bash tools/gate.sh`: `uv sync`, `pytest -q -n 16`, `validate_rdf.py`,
`scoreboard.py --check`, `git diff --check`, wheel build, and a clean-venv
install smoke. CI runs exactly this script.

## Ontology Assets

`rc:` (`https://richcanopy.org/ns/rc#`) is a small shared kernel for
data-project modelling — assets, tables, columns, value types, caveats,
observations, claims, patterns, evidence, layout, storage access,
provenance — not a closed world. Client/domain terms live in project
namespaces (`ais:`, `pm:`, ...). The bundled fixtures are the ontology's
pressure test; the AIS fixture is deliberately reduced (see
`docs/agent/fixture-notes.md`) and doubles as the query-planning gap
walkthrough.

## Known Limits (Honest, Current)

- Term strings are stored directly (no interned term IDs); blank-node IDs
  import as-is — sufficient for current fixtures, not a merge strategy.
- Staged revisions: conservative conflict checks + exact drift snapshots;
  no semantic merge/rebase, no historical graph checkout/replay beyond the
  stored snapshot timeline.
- Search is lexical-only. Context slices are route-explained first passes,
  not proofs of closure.
- RDFLib emits deprecation warnings for some Dataset/TriG internals during
  tests; harmless, revisit on RDFLib upgrades.

## Direction

Until the Distillation Program exits, doc 07 governs: no new tools, fields,
or doc sections without a corresponding removal. Post-program direction is
decided at the program retrospective; the V2 ideas in
`doxabase_design_docs/04-doxabase-v2-backlog.md` are inputs, not
commitments. The permanent posture survives either way: helpers are
drafting aids, not form workflows — the acceptance test is whether an agent
can stage an awkward systematisation that does not fit existing templates.
