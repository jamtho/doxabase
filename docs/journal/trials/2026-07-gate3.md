# Standard Trial Battery — Gate 3/4 Rerun (2026-07-05)

Like-for-like rerun of the battery against the distilled surface (25 tools,
post-envelope-diet, commit `442dcb6`), compared with the Phase 0 baseline
(`docs/journal/trials/2026-07-baseline.md`; 89 tools). Same tasks, same
rubric, same method (Sonnet cold agents, FastMCP-dispatch bridge, full
tool-registry JSON charged as the schema constant: now 35,136 chars vs
138,353 at baseline).

## Results

| Task | Success | Calls | Errors | Total context cost | Baseline | Δ |
|---|---|---|---|---|---|---|
| T1 lore recall | **yes** | 13 | 0 | ~49.6k tok | ~59.9k | **−17%** |
| T2 record & promote | **yes** (a–e) | 37 | 6* | ~90.7k tok | ~154.4k | **−41%** |

*T2's six "errors" were deliberate empty-spec probe calls — the intended
discovery path for kind-dispatched tools (the agent called the targeted
errors "genuinely well-formed and complete"). Baseline T2 had one
accidental error; zero accidental errors occurred in this rerun.

Success ≥ baseline everywhere; quality visibly higher: T1 additionally
traced the evidence provenance chain (identifying the caveat as
converted-document knowledge, not measured — depth the baseline run never
reached); T3 used `describe_query_context(plan_candidate=...)` exactly as
designed, praised the suggestion carrying the exact candidate selector, and
correctly refused the un-asserted path; T2 completed the full ladder with a
correctly-scoped promotion (one genuine map gap, explicit rationale for the
non-change).

## Gate verdict

Doc 07 expected ≥60% cost reduction at this gate; measured 17–45%. No task
got harder — the shortfall is diagnosed, not explained away:

1. **Docs now dominate cost.** T1 doc reads: 97k chars (44k of it the old
   fat doc list — twice); T2: 198k chars, including 12k-char slices of
   `mcp_tools` (204KB) and `response_shapes` (357KB) that still describe
   the dead 89-tool surface and named tools that no longer exist. T2's #1
   friction: docs contradicting the live registry.
2. Fixed immediately in this wave: the doc *list* shipped full per-section
   byte-offset tables (31k chars) — now anchors only (9.4k).
3. The rest is exactly Phase 6: rewrite topic docs against the 25-tool
   surface, retire/regenerate response-shapes/mcp-tools/api-reference,
   respect per-doc caps. **Rerun the battery after Phase 6**; the ≥60%
   target should be judged there.

## Product signal for later phases

- Caveat → evidence takes a hop through the observations graph; a direct
  pointer would save a call (Phase 7 seed list).
- `record_map_fact` (direct writer) vs the staged flow confused T2 until
  docs clarified; the docs pass must state the rule plainly: direct
  recording is for authoring new facts; staged revisions are for reviewed
  changes to existing facts.
- kind/spec tools are discoverable via targeted errors (works, by design),
  but the per-kind spec fields belong in the regenerated tool doc so probe
  calls become optional.
- `query_planning`'s load-bearing "AIS DailyIndex gap" section survived
  only because it sits early in a 53KB doc; Phase 6 caps fix this class.
